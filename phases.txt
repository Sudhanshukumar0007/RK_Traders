1. Order number concurrency — the fallback shouldn't be a "fallback," it should be the primary design.
The plan says "count orders for today + 1, and if that has concurrency gaps, fall back to a sequence." In practice, count-then-insert under concurrent requests will produce duplicate order numbers — two requests can both count "5 orders today" and both generate SHP-20260615-0006. This isn't an edge case that occasionally appears; it's the default failure mode under any real traffic.
Fix: use a PostgreSQL sequence (or a SELECT ... FOR UPDATE on a per-day counter row) from day one. A common pattern is a daily_order_counters table with (date, last_number) and an atomic UPDATE ... SET last_number = last_number + 1 RETURNING last_number, wrapped in the same transaction as order creation. Don't ship the naive count-based version even temporarily — retrofitting this after orders exist means reconciling potential duplicate order numbers in production data.
2. Wholesale pricing trigger — "OR" needs a tie-breaker.
The plan states wholesale pricing applies if either the user is is_wholesale_customer or quantity meets wholesale_min_qty. That's reasonable, but it means a retail customer ordering 1 unit of an item where wholesale_min_qty = 1 (or null/0, if any seeded products have that) would silently get wholesale pricing. Worth explicitly deciding: should wholesale_min_qty ever be allowed to be 1 or unset, and if a wholesale customer orders below the box quantity, do they get wholesale price anyway (because they're a verified wholesale account) or only the per-unit MRP? The plan as written gives wholesale customers wholesale pricing regardless of quantity — confirm that's actually the intended business rule, since it affects the price snapshot logic and is hard to silently fix later without reprocessing historical orders.
Everything else is solid:

HTTP-only cookie for guest cart session — correct choice, avoids XSS token theft.
Cart merge clearing the guest cart and summing quantities on conflict — sensible default.
Price/SKU/size-label snapshotting on order creation — correctly avoids the "live price reference" trap flagged earlier.
Stock decrement tied to status transition (not order creation) — correct, matches the Phase 6 reservation design that's coming later. Just make sure the transition logic is idempotent (checking "if not already decremented" as stated) — that's already called out, good.
Test coverage looks comprehensive for the stated scope.

One omission, not blocking but worth a follow-up note: the flat-rate INR 100 shipping is a known placeholder (Phase 5 replaces it), but make sure the Order model's shipping_cost field isn't hardcoded into the order-creation logic in a way that's awkward to swap out — i.e., it should be assigned from a calculate_shipping(order) function stub that returns 100 for now, so Phase 5 only needs to change the function body, not the call sites.
With #1 fixed (non-negotiable) and #2 confirmed as a deliberate business decision (not an oversight), this is ready to move on.
